# IMX 1.0 hardening audit

Audit date: 2026-05-28 (Pacific/Honolulu).

Base commit before this audit branch: `c8b2bf4` (current `main` at branch
creation).

This document reconciles the local high-bar backlog with the current tree. It
is intentionally evidence-based: items are marked closed only when the current
repo contains a script, test, or CI step that proves the claim.

## Closed by current evidence

- Decode fuzz coverage is complete for the current codec set. There are nine
  codec crates under `crates/codecs/` and nine matching fuzz targets:
  `bmp_decode`, `farbfeld_decode`, `gif_decode`, `jpeg_decode`, `png_decode`,
  `pnm_decode`, `qoi_decode`, `tiff_decode`, and `webp_decode`.
- The fuzz smoke gate is enforced by `scripts/ci.sh`, which runs
  `scripts/run-fuzz.sh`; the standalone preview workflow runs that gate on
  Ubuntu pull requests and main pushes.
- Benchmark coverage exists for decode/encode/transcode hot paths plus nearest
  and filtered resize (`box`, `triangle`, `catmull-rom`, `lanczos3`) through
  `benches/throughput.rs`, `scripts/bench-release.sh`, and
  `scripts/bench-regression.sh`.
- Visual regression coverage now includes fixed golden pixels for every resize
  filter, representative color/tone operations, and a geometry transform chain
  in `tests/visual_regression.rs`.

## Added hardening

- The workspace declares Rust MSRV `1.85`.
- `scripts/check-msrv.sh` checks the whole workspace with the MSRV toolchain.
- `.github/workflows/rust-standalone-preview.yml` installs Rust `1.85.0` and
  runs the MSRV check on Ubuntu without hosted Apple runners.
- `ImageError` is `#[non_exhaustive]` and documents stable diagnostic-code use
  for 1.x callers.
- `docs/api-freeze.md` defines the pre-tag API surface review and the
  `cargo-public-api` snapshot command for `imx-core`.

## Still operator-gated

- Publishing crates, cutting tags, pushing branches, and GitHub-hosted release
  jobs are outside this audit. They remain manual/operator actions.
- `cargo public-api` is a review tool, not a shipped dependency; install it
  locally before a release candidate and store the snapshot under `target/`.
- The current repository still needs real release-candidate evidence from the
  full local/hosted release gates before claiming a 1.0 tag is ready.
