# IMX 1.0 readiness

This document makes the case that the `imx` workspace is ready to commit to a
**1.0** release with a real stability guarantee, and spells out exactly what
that guarantee does and does not cover. It is a decision document, not a
version bump: **no version number is changed here.** Bumping
`workspace.package.version` is a deliberate manual step (see
[releasing.md](releasing.md) and [`RELEASE_CHECKLIST.md`](../RELEASE_CHECKLIST.md)).

## Why IMX can stand behind a 1.0

The 1.0 promise rests on properties that are *enforced*, not aspirational:

- **Determinism.** The same input plus the same arguments always produces
  byte-identical output. Encoders use fixed parameters and stable orderings;
  output is written via a temp file plus atomic rename, so a malformed input
  never leaves a partial file behind. Determinism is what makes IMX safe to put
  in build pipelines, caches, and content-addressed stores.
- **Differential verification against ImageMagick.** Identify/decode/transcode
  behavior is checked against a mature reference implementation used as an
  external oracle (`IMX_REQUIRE_ORACLE=1` in CI), so correctness is anchored to
  something other than IMX's own expectations.
- **Coverage-guided fuzzing.** Per-codec fuzz targets exercise the decode and
  identify entrypoints (BMP, FARBFELD, GIF, JPEG, PNG, PNM, QOI, WebP, and —
  in this release — TIFF), backed by seeded corpora and a malformed-input
  suite, with crash artifacts retained from scheduled long-running runs.
- **Bounded, checked allocation.** Decoded pixel buffers are capped
  (`MAX_PIXEL_BYTES` = 512 MiB; JPEG decode capped at 128 MiB for decoder
  overhead) and every allocation goes through `try_reserve_exact`, so hostile
  dimensions return an error instead of aborting. CLI input reads are capped at
  513 MiB.
- **Memory safety by construction.** The product decode/encode paths are safe
  Rust; runtime dependencies are the local IMX crates plus pure-Rust codecs.

Together these mean the surface IMX exposes is not just *implemented* but
*verified*, *bounded*, and *reproducible* — the preconditions for a version a
caller can pin and trust.

## What a 1.0 makes stable

The following are **contracts** under 1.0 semver — breaking them requires a
2.0:

- **CLI command and option grammar.** The subcommands and flags enumerated by
  `imx --help` (identify, report, compare, resize, resize-fit, crop, rotate,
  flip, flop, pipeline, batch-convert, assemble, completions, self-test, and
  the bare transcode form) and their argument shapes.
- **Exit codes.** `0` success, `1` operation/IO/data error, `2` usage error.
- **`identify --json` schema** (`schema_version` 1: format/width/height/
  channels/depth).
- **`report --json` schema** (`schema_version` 2: the identify fields plus
  `status`, `diagnostic_code`, and `frames`). The set of stable
  `diagnostic_code` values is a contract; new codes may be added, but existing
  codes will not change meaning within 1.x.
- **Deterministic byte output** for a given format, filter, and quality. A
  same-format rewrite is a documented decode/re-encode and does not preserve
  incidental source representation (whitespace, comment bytes, opcode choices).
- **Format support matrix** as published in the README: which formats decode,
  which encode, and the documented lossy conversions.

## What stays explicitly fluid in 1.x

These may change without a major bump, and callers should not pin to them:

- **The exact bytes of a *new* encoder or a *changed default*** (e.g. adding a
  new `--filter` kernel, or output of a format IMX does not yet support).
  Output is deterministic for a fixed set of inputs+flags, but adding formats
  or filters is additive, not a break.
- **Internal crate APIs of the codec crates.** Only `imx-core`'s documented
  public API is treated as a stable library contract. The `imx-codec-*` crates
  are published so the CLI resolves, but their internals may move within 1.x.
- **Performance and memory characteristics**, beyond the documented allocation
  caps.
- **`imx-preview`** — never published; purely an internal harness.

## Gating condition: ship resampling first

**1.0 should be tagged only after the real resampling default lands.** Until
this batch, `resize`/`resize-fit` used nearest-neighbor sampling, which
produces visibly aliased downscales. Tagging 1.0 with nearest-neighbor as the
resize default would make resize quality an embarrassment at the exact moment
the project is asking users to commit to it.

This release introduces `--filter` with **Lanczos3 as the default**, so the
default resize path is high quality. The readiness condition is therefore:

> Tag 1.0 from a commit where `--filter` is present and Lanczos3 is the resize
> default, with the differential corpus green against the ImageMagick oracle.

Once that commit exists and CI is green (including the oracle differential),
follow [`RELEASE_CHECKLIST.md`](../RELEASE_CHECKLIST.md) to bump the version,
publish, and tag.
